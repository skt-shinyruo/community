# Big-bang 切换与回滚演练 Runbook（docker compose）

Directory: `.helloagents/archive/2026-01/202601170935_legacy_cutover_prod_parity/`

> 目标：把 Big-bang 切换从“口头承诺”变成可执行 SOP，确保出现故障时能在可接受时间内回到“可登录/可浏览/可发帖”的可用状态。

---

## 1. 切换前门禁（必须全部通过）

1. 功能门禁：`acceptance-matrix.md` 覆盖 100% 功能，并完成至少 80% 自动化（其余必须有人工回归步骤）
2. 全依赖门禁：Nacos/MySQL/Redis/Kafka/Elasticsearch 均在目标环境启用并通过健康检查
3. 可观测门禁：指标/日志/Trace 可用；关键告警规则已配置
4. 数据门禁：MySQL 全量备份已完成并校验；ES 索引模板与重建脚本可执行；Kafka 保留策略确认
5. 压测门禁：关键接口压测基线达标（p95/p99/错误率/吞吐）

## 2. 切换步骤（示意）

1. 冻结写入窗口（公告/维护窗口）
2. 停止 legacy-community 写入口（只读/下线写操作）
3. 启动/确认新体系全服务健康（gateway + 7 服务）
4. 执行 ES reindex（如需要）并验证搜索
5. 切换入口（DNS/端口/网关路由/前端入口）
6. 观察关键指标（登录成功率、错误率、p95 延迟、Kafka 堆积、ES 查询失败率）
7. 完成灰度观察窗口后解除冻结

## 3. 回滚策略（必须可演练）

回滚触发条件（示例）：
- 登录失败率持续高于阈值
- 错误率/延迟超阈值无法在限定时间内恢复
- 数据一致性问题导致核心功能不可用

回滚步骤（示意）：
1. 重新冻结写入窗口
2. 将入口切回 legacy（DNS/端口/路由）
3. 对新体系停止外部流量，仅保留必要消费/索引重建或停用
4. 若新体系产生写入：按策略回放/补偿或丢弃（以 MySQL 备份为准）
5. 复盘与修复后重新演练

## 4. 灾备与恢复（概要）

- MySQL：以备份为 SSOT，恢复后需要重建 ES 索引；Redis 可清空重建；Kafka 可按保留策略回放
- Elasticsearch：支持全量重建（reindex）并校验映射与高亮
- Redis：可按 key 前缀清理；限流/验证码等为短 TTL，可直接清理
