## 1. 需求

### 背景
用户反馈：项目“看起来不大”，但仓库拆成了较多 Maven 模块（contracts/infra/common/*-api/*-service/gateway 等），担心“过度工程化”带来交付与运维负担。

同时，本项目的目标形态被明确为：**真实微服务**（各服务独立部署/扩容），因此需要从三个维度综合判断“模块数量是否合理”：
- 团队协作/交付：开发效率、联调成本、发布节奏、责任边界
- 技术治理/边界：依赖方向、契约纯度、防止漂移与循环依赖
- 运行与部署：服务数带来的平台/运维复杂度，以及模块划分对部署的实际影响

### 目标
- 解释清楚：**“Maven 模块数”与“运行时服务数”并不等价**，并基于项目现状给出可量化判断。
- 给出结论：当前“模块拆分是否合理”，以及在“真实微服务”目标下的推荐做法。
- 给出三套可选路径（保持现状 / 适度收敛 / 激进拆分），说明适用条件与风险，供后续演进选择。

## 2. 方案

### 技术方案（评审方法）
1. 统计 Maven 模块（根 `pom.xml` 的 `<modules>`）与可部署服务（`spring-boot-maven-plugin` 出现的模块）。
2. 抽查核心模块定位：
   - `contracts-*` 是否保持“契约纯净”（不携带 runtime 依赖与实现）
   - `*-api` 是否保持“RPC 接口/DTO 纯度”（不引入 Spring 组件/实现类）
   - `common`/`infra-*` 是否边界清晰，避免承载业务域逻辑
3. 将“模块数量”与“真实微服务的关键成本项”对齐，给出建议优先级：  
   运行时服务数、平台依赖（Nacos/Dubbo/Kafka/ES）、交付流程与治理约束，通常比“模块数本身”更决定成本。

### 现状画像（关键事实）
- **Maven 模块数：20 个**（见根 `pom.xml` 的 `<modules>`）。
- **可部署 Spring Boot 服务：9 个**（见 `*/pom.xml` 中 `spring-boot-maven-plugin`）：  
  `gateway`、`ops-service`、`auth-service`、`user-service`、`content-service`、`social-service`、`message-service`、`search-service`、`analytics-service`
- **非部署型库模块：11 个**（契约/基础设施/复用库）：  
  `contracts-core`、`contracts-event-core`、`infra-dubbo-starter`、`infra-security-starter`、`infra-outbox`、`common`、`user-api`、`content-api`、`social-api`、`search-api`、`analytics-api`
- **后端 Java 规模：448 个 `.java` 文件 / 31,238 行**（统计口径：全仓 `.java`）。
- **结构不一致提示：**存在 `message-api/` 目录但缺失 `pom.xml` 且 `src/` 为空，属于“悬挂模块/残留目录”。

### 结论（回答“是否合理”）
在你明确选择的目标（**真实微服务：独立部署/扩容**）下：  
**结论是：总体合理，而且模块多的主要原因不是“拆得碎”，而是“用模块把边界写进构建系统”。**

更具体地说：
- 真实微服务的主要成本来自 **“9 个可运行服务”** 的交付与运维，而不是 20 个 Maven 模块。
- 多出来的 11 个非部署模块主要是“契约与治理基础设施”，用于降低跨服务耦合、避免实现级依赖与漂移；这对长期交付是正收益。

### 三维度评估

#### 1) 团队协作 / 交付
**合理点：**
- 单仓多模块（同版本号）让“多服务”仍能以统一工程化方式构建，降低多仓库/多版本的协作成本。
- `*-api`/`contracts-*` 的存在能把联调从“对实现细节”转为“对接口契约”，降低跨服务改动的阻塞面。

**真实风险（与模块数关系不大）：**
- 9 个服务意味着开发/测试/部署链路天然更长：需要环境编排、契约兼容、灰度/回滚、联调与回归策略。

**建议：**
- 把交付单元明确为“服务”（而不是“模块”）：CI 按路径/模块增量构建（常用：`mvn -pl :user-service -am`）。
- 为服务与契约建立 owner（CODEOWNERS/目录责任），否则模块越多越难协调。

#### 2) 技术治理 / 边界
**合理点：**
- `contracts-*` 纯度好（示例：`contracts-core/pom.xml` 无 Spring 运行期依赖），有利于稳定契约与跨服务共享。
- `*-api` 依赖极轻（示例：`user-api/pom.xml` 只依赖 `contracts-core`），且未发现 Spring 注解使用，符合“接口/DTO jar”定位。
- 没有发现 service→service 的 Maven 直接依赖（实现级耦合风险可控）。

**风险点：**
- `common` 边界易膨胀：短期复用方便，但长期可能变成“上帝模块”，把业务逻辑与领域概念误下沉。
- `infra-*` 的定位需要持续明确：是“只提供 filter/auto-config（provided）”，还是“统一引入全套依赖”，否则会重复依赖与漂移。

**建议：**
- 给 `common` 建立准入清单（允许：Result/错误码/trace/少量通用组件；禁止：业务域逻辑与跨域聚合）。
- 保持 `*-api` 的“纯接口/DTO”纪律：避免引入 Spring、避免把“内部实现 DTO”泄漏为跨服务契约。

#### 3) 运行与部署
**关键点：**
- Maven 模块划分几乎不增加运行复杂度；运行复杂度主要取决于服务数、依赖组件（Nacos/Kafka/ES/Redis/MySQL）与可观测性/弹性治理。
- 当前结构用 `infra-*` 把 trace/security/outbox 等横切能力“收敛到 SSOT”，对运维一致性是加分项（避免每个服务各搞一套）。

**建议：**
- 只要坚持“独立部署/扩容”，就需要把 SLO/告警/限流/熔断/超时基线平台化；否则“服务多”才是主要痛点。

### 方案对比（R3：多方案对比）

#### 方案A（推荐）：保持现状（20 模块 / 9 服务），做“卫生与边界治理”
**做法：**
- 保持 `contracts-* / infra-* / common / *-api / *-service` 分层不变；
- 小步修复不一致点（如清理 `message-api/` 残留目录）；
- 用文档与门禁守住 `common` 与 `*-api` 的边界。
**优点：**成本最低、与当前架构决策一致、风险可控。  
**缺点：**需要持续治理纪律；如果团队/平台能力不足，9 个服务的运维成本依然存在。

#### 方案B：领域服务收敛（Domain Pods）——“减少服务/模块数量但保留独立部署”
（来源：子代理独立构思，摘要整合）
**做法：**
- 将强耦合的服务按领域收敛为少数“领域服务”，域内做模块化单体（Modulith），对外仍保持微服务独立部署；
- 通过架构约束/契约测试守住域内边界，避免退化成大泥球。
**优点：**服务数减少，运维与联调成本下降；仍保留独立部署。  
**缺点：**单个部署单元爆炸半径变大；需要更强的边界治理工具，否则会快速腐化。
**适用：**团队规模有限、平台资源紧张、服务间同步调用过碎“聊天化”的阶段。

#### 方案C：真正独立版本/独立仓库的微服务（契约即产品）
（来源：子代理独立构思，摘要整合）
**做法：**
- 每个服务独立仓库/独立 SemVer/独立 CI/CD；
- 契约（OpenAPI/AsyncAPI/Protobuf 等）独立发布为制品并做兼容性门禁；消费者只依赖契约与生成 SDK/stub。
**优点：**组织与发布解耦最彻底，长期可扩展性最佳。  
**缺点：**治理成本与平台要求最高（版本碎片化、兼容窗口、契约评审、可观测性平台化）。
**适用：**服务数量持续增长、多人多团队并行、需要高频独立发布的成熟阶段。

## 5. 技术决策

> 本方案涉及的技术决策，归档后成为决策的唯一完整记录

### microservices-module-count-review#D001: 维持当前多模块结构，优先治理边界而非“为减模块而减”
**日期**: 2026-02-25
**状态**: ✅采纳
**背景**: 用户目标是“真实微服务（独立部署/扩容）”。在此目标下，模块数量本身不是主要成本项；相反，契约与边界模块是降低长期耦合与交付摩擦的重要工具。当前结构总体健康，优先做卫生治理与边界守卫最具性价比。
**选项分析**:
| 选项 | 优点 | 缺点 |
|------|------|------|
| A: 保持现状 + 边界治理（推荐） | 低风险、成本低、与现有架构决策一致 | 需要持续纪律；服务数带来的运维成本仍需平台化 |
| B: 领域服务收敛（Domain Pods） | 降低服务/模块数量与联调成本，仍可独立部署 | 爆炸半径变大；边界治理要求更高 |
| C: 真正独立仓库/独立版本微服务 | 组织与发布解耦最佳，长期可扩展 | 平台与治理成本最高，落地门槛高 |
**决策**: 选择方案 A
**理由**: 现阶段最需要解决的是“边界一致性与工程卫生”，而不是“模块数量”。A 能保持真实微服务目标，同时用最小成本持续降低耦合与漂移风险。
**影响**: 后续如要落地，优先级建议为：清理残留目录（如 `message-api/`）、补齐 KB 的模块索引粒度、为 `common`/`*-api` 增加边界门禁。
