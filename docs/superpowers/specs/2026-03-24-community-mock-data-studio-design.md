# Community Mock Data Studio Design Spec

**Date:** 2026-03-24
**Status:** Approved for planning
**Owner:** Codex

---

## 1. 背景

当前仓库的本地数据初始化主要依赖 `deploy/mysql-init/001_create_databases.sh` 与 `deploy/mysql-init/010_schema.sql`：

- 这套机制适合“首次初始化数据库时创建 schema + 基础种子数据”
- 但它依赖 MySQL 容器首次初始化时执行，无法天然满足“每次 `docker compose up` 都自动补齐演示数据”
- 当前默认演示数据规模也非常小，主要只有少量用户和基础枚举，不足以支撑首页、搜索、私信、IM、排行榜、审核、奖励等页面的系统性验证

用户已经明确把目标从“只补一批 SQL”升级为“建立一个独立的 mock 数据生成服务”，并确认以下方向：

1. 需要一个新的独立服务，而不是继续把大量逻辑塞进 `mysql-init`
2. 新服务使用 Node.js 实现，并提供一个简单 UI，让用户手动指定生成数量、场景和参数
3. 服务既要支持“启动后自动补数”，也要支持“手动高级生成”
4. 高级生成允许使用 OpenAI / Codex 能力增强文本内容，但自动补数不能依赖 AI 凭据
5. 所有生成数据都必须带批次标识，支持历史查询和按批次删除

本设计的目标，是在不污染现有业务服务边界的前提下，为本仓库增加一个可控、可回收、可扩展的本地演示数据控制平面。

---

## 2. 目标

### 2.1 核心目标

1. 新增独立的 `mock-data-studio` 服务，作为本地 dev/demo 环境的数据生成控制平面
2. 在 `docker compose up` 后自动执行“幂等补数”，让系统始终具备足够可见的演示数据
3. 提供本地 UI，允许用户手动指定用户、帖子、评论、IM、通知、奖励等数据规模
4. 提供“规则生成”与“高级 AI 文案增强”两种生成层级
5. 所有生成结果必须可追踪到批次，且支持按批次删除
6. 让首页、帖子详情、搜索、私信、IM、排行榜、成长中心、审核后台、奖励后台等主要页面在本地 compose 环境下可直接验证

### 2.2 运维与验证目标

- 自动补数不依赖 `OPENAI_API_KEY` 或其他 AI 凭据
- 手动高级生成只有在环境变量完整且用户显式启用时才调用 AI
- 生成失败时必须能看到失败阶段、错误摘要和已完成数量
- 第一版只允许单任务串行执行，避免多个批次并发污染库状态

### 2.3 非目标

本轮设计不包括以下能力：

- 生产环境开放该服务
- 将该服务接入现有正式登录体系
- 断点续跑、多任务并发调度、分布式任务执行
- 用 AI 决定主键、关系、删除链路或数据库写入策略
- 将所有数据都强制通过现有业务 API 生成

## 3. 已确认产品方向

以下方向已经由用户明确确认：

- 服务形态：单个长驻服务，同时承担自动补数与手动生成
- 部署方式：独立服务、独立端口，不挂进现有 `frontend/`
- 环境约束：仅 `dev/demo` 启用，仅绑定宿主机 `127.0.0.1`
- 启动模式：支持“自动补数”与“手动高级生成”并存
- 写入策略：采用混合模式
  - 大批量基础数据直写数据库
  - 少量必须走真实业务副作用的动作走业务 API
- AI 使用边界：仅用于手动高级生成，且必须依赖环境变量提供必要参数
- 批次管理：所有生成数据必须带 `batch` 标识，并支持按批次删除
- 第一版执行模型：单任务串行，失败后优先按批次删除后重跑

## 4. 现有系统事实与约束

### 4.1 compose 与本地管理面事实

当前 `deploy/docker-compose.yml` 已经体现了本地控制面服务的几条约束：

- 业务主入口为 `community-gateway` `12880` 与前端 `12881`
- MailHog、XXL-Job Admin、Grafana 等辅助 UI 都是 localhost-only 暴露
- `debug` 与 `observability` profile 已经建立了“本地调试/观测能力单独暴露”的模式

这意味着 `mock-data-studio` 最合理的落点是：

- 作为 `deploy/docker-compose.yml` 中的新独立服务
- 仅暴露到 `127.0.0.1`
- 用单独 profile 或环境变量控制启停

### 4.2 当前数据初始化事实

当前本地数据库初始化主要依赖：

- `deploy/mysql-init/001_create_databases.sh`
- `deploy/mysql-init/010_schema.sql`

这套机制只在 MySQL 数据卷首次初始化时执行，适合做：

- schema 初始化
- 极小规模基础 seed

不适合做：

- 每次启动都补齐数据
- 可配置数量的批量生成
- 带任务状态和批次回收的生成流程

### 4.3 当前前端验证面事实

现有前端路由已经覆盖大量需要演示数据的页面，例如：

- `PostsView` / `PostDetailView`
- `SearchView`
- `ConversationsView` / `ConversationDetailView`
- `NoticesView`
- `GrowthCenterView` / `TaskCenterView` / `LeaderboardView`
- `ModerationView`
- `RewardOpsView` / `GrowthAdminView`

因此这个项目的痛点不是“没有页面”，而是“页面没有足够真实的数据可看”。

### 4.4 OpenAI / Codex 使用事实

用户希望新服务使用 Node.js，并在“手动高级生成”中使用 Codex 能力来增强 mock 内容质量。

本设计将这一能力限定为：

- 使用 Node 侧 OpenAI SDK 与官方 API
- 只服务于文本内容增强
- 不承担数量控制、关系生成、批次管理或数据库写入职责

这是为了避免把生成稳定性、成本和结构一致性绑死在 AI 推理上。

## 5. 方案对比

### 5.1 方案 A：继续扩展 `mysql-init`

做法：

- 继续在 `deploy/mysql-init/010_schema.sql` 中补更多种子数据

优点：

- 实现简单

缺点：

- 只在新 volume 初始化时生效
- 无法支持每次启动自动补数
- 无法提供 UI 和批次管理

### 5.2 方案 B：把生成器塞进 `community-app`

做法：

- 在 Spring Boot 启动期或内部运维接口中实现 mock 数据生成

优点：

- 可以直接复用部分业务模块

缺点：

- 与业务服务启动强耦合
- 不适合混合写 `community` 与 `im_core`
- Node + Codex 的使用边界会变得别扭

### 5.3 方案 C：独立 `mock-data-studio` 服务

做法：

- 新增一个独立 Node 服务，提供 UI、任务执行、批次管理与自动补数能力

优点：

- 与主业务服务解耦
- 同时适配自动补数和手动高级生成
- 更适合承载 Node SDK、任务状态、批次删除等能力
- 可以直接连接 `community` / `im_core` 数据库，并按需要调用现有业务 API

缺点：

- 需要新增一套服务目录、compose 配置和批次跟踪表

### 5.4 推荐方案

采用方案 C。

原因：

- 它最符合用户已经确认的“独立服务 + UI + 自动补数 + 手动高级生成”目标
- 它与当前 compose 的 localhost-only 管理面模式一致
- 它最容易控制 AI 使用边界和批次回收能力

## 6. 总体设计

### 6.1 服务边界

新增一个独立服务：`mock-data-studio`。

职责包括：

- 提供本地 Web UI
- 启动时执行自动补数检查
- 接收手动生成请求
- 维护任务状态与批次历史
- 协调规则生成器、AI 文案增强器、数据库写入器、API 写入器
- 在生成结束后触发搜索/派生数据补建

不负责：

- 正式业务请求流量
- 正式登录与鉴权
- 线上环境数据管理

### 6.2 两种运行模式

#### 6.2.1 自动补数模式

服务启动后读取默认目标规模，检查“默认批次”当前已有数量，只补缺口。

示例：

- 目标帖子数 = 5000
- 当前默认批次已有帖子 = 4200
- 本次只补 800

自动补数模式特点：

- 不依赖 AI 凭据
- 只作用于默认批次
- 目标是保证系统总有足够演示数据，而不是每次重建全部数据

#### 6.2.2 手动高级生成模式

用户通过 UI 显式提交任务，填写：

- 场景预设
- 各类数据数量
- 是否启用 AI 文案增强
- 批次名称或备注

手动模式可以创建独立批次，不影响默认批次。

### 6.3 推荐服务目录

推荐新增独立目录，例如：

- `tools/mock-data-studio/`

其中包含：

- `src/server/`：HTTP 服务与任务 API
- `src/ui/`：轻量前端页面
- `src/jobs/`：任务执行与状态机
- `src/generator/`：规则生成器与场景模板
- `src/ai/`：OpenAI / Codex 文案增强封装
- `src/writers/`：数据库写入与 API 写入
- `src/batches/`：批次统计、删除、历史查询
- `src/config/`：环境变量与运行时配置

## 7. 数据模型与批次管理

### 7.1 跟踪表

推荐增加以下内部管理表：

#### 7.1.1 `demo_batch`

表示一次生成批次。

建议字段：

- `id`
- `batch_key`
- `batch_type`：`default` / `manual`
- `scene`
- `source_mode`：`auto-fill` / `manual-rule` / `manual-ai`
- `status`
- `notes`
- `created_at`
- `updated_at`

#### 7.1.2 `demo_job`

表示一次执行任务。

建议字段：

- `id`
- `job_key`
- `batch_id`
- `trigger_type`：`startup` / `manual`
- `parameters_json`
- `status`
- `phase`
- `error_summary`
- `started_at`
- `finished_at`

#### 7.1.3 `demo_batch_target`

记录该批次的目标数量与实际数量。

建议字段：

- `batch_id`
- `entity_type`
- `target_count`
- `actual_count`
- `failed_count`

### 7.2 业务数据的批次标记

对受 `mock-data-studio` 管理的数据，需要支持按批次追踪和删除。

推荐做法：

- 对可控业务表增加 `demo_batch_id` 与 `demo_source`
- 如果某些表不适合直接加字段，则增加映射表记录实体与批次关系

第一版不追求“所有表统一一种技术方案”，重点是：

- 能可靠统计
- 能可靠删除
- 不污染现有正式业务逻辑

### 7.3 默认批次与手动批次

- 默认批次：自动补数专用，长期存在
- 手动批次：每次手动生成创建独立批次

这样可以避免：

- 自动补数和手动试验互相覆盖
- 删除某个手动批次时误删默认演示数据

## 8. 生成引擎与写入策略

### 8.1 场景预设

建议第一版内置若干场景预设：

1. 技术社区热启动
2. 审核压力场景
3. IM 活跃场景
4. 奖励兑换活跃场景

预设负责定义：

- 用户活跃度分层
- 帖子/评论/点赞/关注分布
- 私信与群聊密度
- 举报、通知、奖励订单的出现概率

### 8.2 规则生成器

规则生成器负责：

- 用户画像与关系图谱
- 分类、标签、帖子和评论分布
- 互动强弱、时间分布、消息节奏
- 审核样本、奖励订单样本、通知样本

规则生成器必须是默认路径，且不依赖 AI。

### 8.3 AI 文案增强器

AI 只在用户显式启用“高级 AI 文案增强”时参与。

作用范围：

- 帖子标题与正文
- 评论与回复文本
- 通知标题与正文
- 部分 IM 私聊或群聊内容

AI 不参与：

- 数量控制
- 主键生成
- 关系建模
- 批次管理
- 删除逻辑
- 数据库写入

### 8.4 混合写入策略

#### 8.4.1 直写数据库

适合：

- 用户基础档案
- 帖子/评论主体数据
- 社交关系基础数据
- IM 历史消息与房间数据
- 成长/奖励/审核等大批量样本数据

原因：

- 快
- 更适合大规模生成
- 可控性更高

#### 8.4.2 调用业务 API

只用于少量“必须保留真实副作用语义”的动作。

例如：

- 需要走现有业务校验和投影链路的特定样本操作
- 少量需要验证端到端行为的示例动作

第一版不追求“所有数据都经由业务 API”。

## 9. Codex / OpenAI 集成设计

### 9.1 集成原则

Node 服务中的 OpenAI 能力只作为“高级文本生成器”存在。

推荐采用：

- Node 侧 OpenAI SDK
- 官方 API 调用封装为单独 `ai` 模块

### 9.2 启用条件

只有在以下条件同时满足时才允许启用：

1. 用户在 UI 中勾选高级 AI 文案增强
2. 环境变量完整
3. 本次任务规模没有超过高级模式允许上限

### 9.3 环境变量

第一版至少需要：

- OpenAI 凭据相关环境变量
- 模型名
- 超时、并发和预算上限配置

这些值通过 `env` 提供，不允许在 UI 中直接输入秘密信息。

### 9.4 风险控制

高级 AI 模式需要明确限制：

- 单次请求文本数量
- 单批次允许的 AI 内容占比
- 超时时间
- 并发数

否则大规模批次会带来明显的时延和成本问题。

## 10. UI 设计

### 10.1 页面结构

推荐 UI 至少包含以下页面：

1. 生成数据
2. 场景预设
3. 批次历史
4. 批次详情
5. 系统状态

### 10.2 生成数据页

展示并允许编辑：

- 模式：自动补数 / 手动生成
- 场景预设
- 用户、帖子、评论、点赞、关注、私信、群聊、通知、举报、订单数量
- 规则生成 / AI 增强开关
- 预估摘要
- 开始生成 / 停止任务

### 10.3 批次历史页

展示：

- 默认批次与手动批次
- 创建时间、模式、场景、状态、耗时
- 目标数量与实际数量摘要
- 删除入口

### 10.4 批次详情页

展示：

- 各类实体目标值 / 实际值 / 失败值
- 当前任务阶段
- 错误摘要
- 搜索补建状态
- 删除与重跑建议

### 10.5 系统状态页

展示：

- MySQL / `im_core` 连接状态
- `community-app` / gateway API 连通性
- OpenAI / Codex 相关环境变量状态
- 当前是否存在运行中的任务

## 11. 执行流与失败处理

### 11.1 执行阶段

推荐执行流为：

1. 接收任务并落 `demo_job`
2. 校验环境和参数
3. 生成计划
4. 分阶段写入：
   - 用户
   - 内容
   - 社交
   - 消息 / 通知
   - IM
   - 审核 / 奖励
5. 补建派生数据：
   - 搜索重建
   - 其他必要统计刷新
6. 更新批次统计并完成任务

### 11.2 失败处理

第一版采用以下策略：

- 前置校验失败：直接失败，不开始写入
- 阶段执行失败：记录错误与已完成数量，任务状态变为 `PARTIAL_FAILED` 或 `FAILED`
- 不做断点续跑
- 用户优先通过“按批次删除后重跑”恢复

### 11.3 并发策略

第一版只允许一个生成任务运行。

原因：

- 降低批次交叉污染风险
- 降低数据库与 API 压力控制复杂度
- 让任务状态和错误定位更清晰

## 12. compose 与安全约束

### 12.1 compose 集成

`mock-data-studio` 推荐作为新的 compose 服务加入 `deploy/docker-compose.yml`。

建议：

- 只绑定 `127.0.0.1`
- 有单独端口，例如 `12888`
- 依赖 MySQL、`community-app`、`im-core` 等核心服务可用
- 用环境变量控制自动补数和 AI 能力

### 12.2 环境限制

该服务应明确为：

- `dev/demo` 专用
- `prod` 默认不启动
- 不接入正式用户登录体系

原因：

- 它是高风险数据控制面
- 它的主要目标是本地验证与开发联调

## 13. 验证与验收

### 13.1 服务级验收

应满足：

- `mock-data-studio` 可正常启动
- UI 可访问
- 系统状态页能正确展示依赖连通性
- 自动补数可在启动后触发

### 13.2 数据级验收

应满足：

- 默认批次可按目标值自动补齐缺口
- 手动批次可生成独立数据
- 批次详情可查看数量、阶段、错误、耗时
- 批次删除后，对应数据被可靠清理

### 13.3 页面级验收

在本地 compose 环境中，生成完成后至少应能直接看到：

- 首页有足够帖子流
- 帖子详情有评论/回复
- 搜索页能搜到内容
- 私信 / IM 页面有历史内容
- 排行榜和成长中心有可见数据
- 审核后台有样本
- 奖励后台有商品/订单/指标样本

## 14. 分期建议

### Phase 1：最小可用 Studio

- 独立 Node 服务与 localhost-only UI
- 默认批次与手动批次
- 规则生成器
- 批次管理、删除与状态查看
- 覆盖用户 / 帖子 / 评论 / 社交 / 搜索可见数据

### Phase 2：扩展消息、IM 与后台样本

- 站内信 / 通知
- IM 房间 / 私聊历史
- 审核、举报、奖励、排行榜样本

### Phase 3：高级 AI 文案增强

- Node 侧 OpenAI / Codex 集成
- UI 环境检查
- 高级生成上限与成本控制

---

## 15. 结论

本设计将“本地演示数据初始化”从一次性 SQL 种子升级为一个独立的、本地专用的数据控制平面：

- `mock-data-studio` 作为独立 Node 服务运行
- 自动补数保证 `docker compose up` 后始终有数据可看
- 手动高级生成允许用户按场景和规模造数
- AI 只参与文案增强，不进入结构和写库控制面
- 批次与任务模型保证数据可追踪、可统计、可删除

这条路线比“继续堆 mysql-init”更符合当前需求，也比“把生成器塞进业务服务”更清晰、可维护。

下一步应基于本设计输出 implementation plan，明确目录结构、compose 接入、批次表结构、任务 API、UI 页面、数据库写入顺序、以及 OpenAI / Codex 集成的具体接口。
