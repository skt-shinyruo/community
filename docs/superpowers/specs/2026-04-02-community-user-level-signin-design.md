# Community User Level By Sign-In Design Spec

**Date:** 2026-04-02
**Status:** Approved for planning
**Owner:** Codex

---

## 1. 背景

当前项目已经具备签到能力、成长中心和管理员成长后台：

- `growth_check_in` 已记录用户按自然日签到的事实
- `CheckInService` 已处理每日签到写入与状态查询
- `/api/growth/summary` 已提供成长中心概要数据
- `/api/growth/admin/**` 与前端 `GrowthAdminView` 已形成后台配置与管理的基本模式

但当前项目中的用户 `level` 仍然来自 `user.score`，本质上是积分等级，而不是用户独立等级。这带来两个问题：

1. 用户等级语义和成长值语义混在一起，难以解释
2. 如果想引入“签到型用户等级”，现有 `score -> level` 逻辑无法直接复用

本设计的目标，是在不推翻现有成长值体系的前提下，新增一套独立的用户等级系统。该等级仅由最近一段时间内的签到表现决定，并通过后台阈值配置控制升级门槛。

---

## 2. 目标

### 2.1 核心目标

1. 新增一套独立于 `user.score` 的用户等级体系
2. 用户等级按“最近 100 天签到次数”判定，而不是按积分推导
3. 等级只有 `LV1`、`LV2`、`LV3` 三档，`LV3` 为最高等级
4. 后台可配置等级阈值，但不引入通用规则引擎或表达式系统
5. 用户侧在成长中心和用户主页可看到新等级及最近窗口签到天数

### 2.2 业务目标

- 给用户提供一个简单、清晰、可解释的签到型身份标识
- 避免“断签一天就大幅掉级”的激进连续签到逻辑
- 让运营可以通过后台调整门槛，而不需要改代码或改 SQL

### 2.3 非目标

本设计不包括以下内容：

- 基于发帖、评论、点赞、收藏、举报命中等多维度的复杂等级系统
- 通用规则编排器、脚本化规则引擎、可视化表达式配置
- 用户等级权益体系、专属权限、头像框、称号、徽章
- 排行榜排序逻辑调整
- 彻底移除旧的 `score -> level` 展示链路

---

## 3. 已确认产品方向

以下内容在本轮设计中视为确认结论：

- 用户只保留一个总等级，不拆分多维等级
- 等级不再使用连续签到逻辑
- 等级规则改为“最近 100 天内签到次数”
- 等级档位固定为：
  - `LV1`
  - `LV2`
  - `LV3`
- 默认阈值固定为：
  - 最近 `100` 天签到 `< 12` 天：`LV1`
  - 最近 `100` 天签到 `12-87` 天：`LV2`
  - 最近 `100` 天签到 `>= 88` 天：`LV3`
- 后台允许调整窗口与阈值，但不支持自由组合规则
- 用户认可这是“滚动窗口自然重算”，因此允许等级自然升降

## 4. 现有系统事实与约束

### 4.1 已存在的签到事实表

当前项目已存在 `growth_check_in` 表：

- 每个用户、每个自然日最多一条签到记录
- 表上已有 `(user_id, biz_date)` 相关唯一约束和索引
- 该表足以支持“最近 N 天签到次数统计”

这意味着：

- 本设计不需要新增签到明细表
- 不需要再为用户等级引入第二套签到存储

### 4.2 已存在的签到服务

当前 `CheckInService` 已具备以下能力：

- 按自然日处理签到幂等
- 查询当天是否已签到
- 查询累计签到天数、最大连续签到天数、当前连续签到天数
- 查询月度签到日历

这意味着：

- 用户等级不需要改造签到写链路
- 只需新增一个独立的等级读取与配置能力

### 4.3 当前用户等级与成长值耦合

当前用户 `level` 来自 `score`：

- `PointsService.levelForScore(score)`
- `UserQueryService` 在用户资料视图中回填 `level`
- `GrowthController.summary` 返回 `score` 和 `level`
- `LeaderboardService` 也消费 `score` 与 `level`

这意味着：

- 新等级不能直接复用现有 `level` 字段，否则会污染旧语义
- 一期应新增 `userLevel` 之类的独立响应字段
- 旧 `level` 先保留，避免一次改动过大

### 4.4 已存在的后台模式

当前项目已经有成长后台：

- `/api/growth/admin/**` 已被 `ADMIN` 权限保护
- `AdminGrowthController` 已承载后台查询和调整接口
- 前端 `GrowthAdminView` 已具备管理表单、列表和交互模式

这意味着：

- 等级规则配置应落在同一后台域
- 不应另起一套独立后台技术栈或页面风格

## 5. 总体设计原则

### 5.1 独立语义原则

用户等级是独立的产品概念，不再由成长值派生。成长值继续保留原有语义，服务现有榜单、积分展示和成长资产。

### 5.2 事实优先原则

等级判定基于签到事实表实时计算，而不是靠额外同步任务生成中间状态。对于简单规则，直接读取事实数据比维护快照更稳。

### 5.3 固定规则原则

后台只允许编辑少量固定参数：

- `window_days`
- `lv2_sign_in_days`
- `lv3_sign_in_days`
- `enabled`

后台不允许：

- 新增自定义指标
- 写逻辑表达式
- 组合 AND / OR 条件
- 发布多版本灰度规则

### 5.4 渐进替换原则

一期新增独立 `userLevel` 字段，但不强制替换所有旧 `level` 消费点。先在成长中心、用户主页、后台用户查询中接入，后续再决定是否替换排行榜等旧展示。

## 6. 等级规则设计

### 6.1 默认规则

默认配置如下：

| 参数 | 默认值 | 说明 |
| --- | --- | --- |
| `window_days` | `100` | 滚动签到统计窗口，按自然日计算，包含当天 |
| `lv2_sign_in_days` | `12` | 达到 `LV2` 所需最近窗口签到天数 |
| `lv3_sign_in_days` | `88` | 达到 `LV3` 所需最近窗口签到天数 |

### 6.2 等级判定

用户等级按最近 `window_days` 天签到次数计算：

- 如果签到次数 `< lv2_sign_in_days`，则为 `LV1`
- 如果签到次数 `>= lv2_sign_in_days` 且 `< lv3_sign_in_days`，则为 `LV2`
- 如果签到次数 `>= lv3_sign_in_days`，则为 `LV3`

### 6.3 时间窗口语义

- “最近 100 天”按自然日窗口计算
- 包含当天业务日
- 同一天签到多次仍只记 1 天
- 窗口滑动导致等级上升或下降属于正常重算，不视为惩罚

### 6.4 示例

- 最近 100 天签到 `0` 天 -> `LV1`
- 最近 100 天签到 `11` 天 -> `LV1`
- 最近 100 天签到 `12` 天 -> `LV2`
- 最近 100 天签到 `87` 天 -> `LV2`
- 最近 100 天签到 `88` 天 -> `LV3`
- 最近 100 天签到 `100` 天 -> `LV3`

## 7. 数据模型

### 7.1 复用现有签到表

继续使用现有 `growth_check_in` 作为用户等级判定事实来源，不新增额外签到明细表。

### 7.2 新增等级规则配置表

建议新增表：`user_level_rule_config`

推荐字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | bigint | 主键 |
| `window_days` | int | 滚动窗口天数 |
| `lv2_sign_in_days` | int | `LV2` 阈值 |
| `lv3_sign_in_days` | int | `LV3` 阈值 |
| `enabled` | tinyint | 是否启用 |
| `updated_by` | int | 最近更新人 |
| `update_time` | timestamp | 最近更新时间 |

### 7.3 不新增用户等级快照表

一期不新增 `user_level_snapshot`、`user_level_history` 一类表。

原因：

- 规则简单，现场计算成本低
- 避免配置变更后快照与事实不一致
- 避免引入额外同步和修复逻辑

如果未来等级规则变复杂或读取量显著上升，再评估是否引入快照层。

## 8. 后端设计

### 8.1 新增服务

建议新增 `UserLevelService`，职责如下：

- 读取当前生效等级配置
- 统计用户最近窗口内签到天数
- 根据规则返回 `userLevel`
- 返回用户等级概要数据，供控制器复用

建议接口形态：

- `UserLevelConfig getActiveConfig()`
- `int countSignInDaysInWindow(int userId, LocalDate bizDate)`
- `UserLevelSummary evaluateLevel(int userId, LocalDate bizDate)`

其中 `UserLevelSummary` 至少包含：

- `userLevel`
- `signInDaysInWindow`
- `windowDays`
- `lv2Threshold`
- `lv3Threshold`

### 8.2 Mapper 扩展

为 `GrowthCheckInMapper` 新增窗口计数查询，例如：

- `countByUserIdBetweenDates(userId, startDate, endDate)`

SQL 基于现有 `(user_id, biz_date)` 范围查询即可。

### 8.3 配置读取策略

系统优先读取 `user_level_rule_config` 中当前启用配置。

如果没有有效配置，则使用默认值：

- `window_days = 100`
- `lv2_sign_in_days = 12`
- `lv3_sign_in_days = 88`
- `enabled = true`

### 8.4 校验规则

后台保存配置时必须校验：

- `window_days > 0`
- `lv2_sign_in_days > 0`
- `lv3_sign_in_days > 0`
- `lv2_sign_in_days < lv3_sign_in_days`
- `lv3_sign_in_days <= window_days`

不合法配置必须直接拒绝，不允许进入数据库。

## 9. API 设计

### 9.1 用户侧接口

#### 9.1.1 成长中心概要

在现有 `/api/growth/summary` 响应中新增字段：

- `userLevel`
- `signInDaysInWindow`
- `windowDays`

保留旧字段：

- `score`
- `level`
- `rewardBalance`
- `frozenBalance`

这样一期可以并存新旧等级语义，不强行替换现有前端逻辑。

#### 9.1.2 用户主页

在现有用户主页响应中新增字段：

- `userLevel`
- `signInDaysInWindow`

旧的 `level` 先保留，避免兼容问题。

### 9.2 管理后台接口

在 `/api/growth/admin/**` 下新增：

- `GET /api/growth/admin/user-level/config`
- `PUT /api/growth/admin/user-level/config`

返回和提交字段仅包含：

- `windowDays`
- `lv2SignInDays`
- `lv3SignInDays`
- `enabled`

### 9.3 后台权限

沿用现有成长后台权限模型，仅 `ADMIN` 可访问配置接口。

## 10. 前端设计

### 10.1 成长中心

在成长中心展示：

- `用户等级：LVx`
- `最近 100 天已签到 y 天`

该展示优先使用新字段 `userLevel`，而不是旧的积分等级字段。

### 10.2 用户主页

在用户主页增加新等级展示：

- `用户等级：LVx`
- 可选附文案：`最近 100 天签到 y 天`

### 10.3 成长后台

在现有 `GrowthAdminView` 中新增一个“用户等级规则”配置区块：

- 窗口天数输入框
- `LV2` 阈值输入框
- `LV3` 阈值输入框
- 启用开关
- 保存按钮

不做以下能力：

- 规则预览
- 版本回滚
- 灰度发布
- 命中用户样本分析

## 11. 计算时机与运行时行为

### 11.1 计算时机

以下读取场景现场计算用户等级：

- 用户读取成长中心概要时
- 用户读取主页时
- 管理员查询用户成长账户时

签到成功后无需额外写入“当前等级”，只保留签到事实写入。

### 11.2 启停行为

如果 `enabled = false`：

- 用户侧不展示新用户等级
- 后台仍可读取和编辑规则配置
- 不删除已有签到事实

### 11.3 排行榜兼容

一期不改排行榜逻辑。

原因：

- 当前排行榜仍按 `score` 排序
- 如果直接把排行榜上的 `level` 改成签到等级，会导致“排序依据”和“展示等级”语义不一致

因此，本设计明确把排行榜调整排除在一期范围之外。

## 12. 测试策略

### 12.1 单元测试

新增 `UserLevelService` 单测，覆盖边界值：

- `0`
- `11`
- `12`
- `87`
- `88`
- `100`

同时覆盖：

- 无配置时使用默认值
- 非法配置不允许保存

### 12.2 集成测试

覆盖以下链路：

- 用户签到若干天后，`/api/growth/summary` 返回正确 `userLevel`
- 用户主页返回正确 `userLevel`
- 修改后台配置后，读取结果立即按新阈值生效

### 12.3 前端测试

覆盖：

- 成长中心正确展示 `userLevel`
- 后台配置表单对非法阈值给出错误提示
- 后台配置保存成功后页面状态刷新

本设计建议按以下顺序落地：

1. 新增等级配置表与 mapper
2. 新增 `UserLevelService` 和窗口签到计数查询

4. 扩展成长后台配置接口
5. 扩展前端成长中心、用户主页和后台配置页面
6. 补齐测试

## 14. 风险与取舍

### 14.1 不落快照的取舍

现场计算比快照更简单，但会让每次读取多一次窗口计数查询。基于当前规则复杂度和已存在索引，这个取舍是合理的。

### 14.2 新旧等级并存的认知风险

一期会同时存在：

- 旧 `level`：积分等级
- 新 `userLevel`：签到等级

这会带来短期认知成本，但它比一次性替换所有旧字段更稳。后续可在用户体验稳定后逐步收敛命名。

### 14.3 阈值调整导致等级波动

后台调整阈值后，用户等级会立即按新配置重算。这是预期行为，因此后台页面需要明确提示“保存后立即生效”。

---

## 15. 结论

本设计选择了一条刻意收缩后的实现路径：

- 不做复杂多维等级
- 不做连续签到掉级
- 不做通用规则引擎
- 只做一个简单、清晰、可配置的签到型用户等级

通过“最近 100 天签到次数 -> `LV1-LV3`”这套固定规则，项目可以在较低实现成本下获得一套独立于积分的用户等级能力，同时复用现有签到事实表、成长中心和后台管理模式。
