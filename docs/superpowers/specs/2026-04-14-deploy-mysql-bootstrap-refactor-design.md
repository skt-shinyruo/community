# Deploy MySQL Bootstrap Refactor Design Spec

**Date:** 2026-04-14
**Status:** Approved for planning
**Owner:** Codex

---

## 1. Background

The current local MySQL bootstrap assets under `deploy/mysql-init/` have grown into a mixed directory that bundles together three different concerns:

1. MySQL primary first-boot init assets for the main product schemas
2. Nacos-specific bootstrap assets
3. XXL-JOB-specific bootstrap assets

This structure made initial delivery fast, but it now hides real ownership boundaries:

- `deploy/mysql-init/010_schema.sql` is a large combined bundle that contains community schema, IM schema, compatibility DDL, metadata tables, and local seeds
- `deploy/mysql-init/030_nacos_schema.sql` is not actually part of the main MySQL init flow; it is consumed by `nacos-db-bootstrap`
- `deploy/mysql-init/020_xxl_job_schema.sql` and `deploy/mysql-init/021_seed_xxl_job.sh` currently piggyback on the main MySQL container's `/docker-entrypoint-initdb.d` mount, even though they serve XXL-JOB specifically
- `deploy/compose.infra.mock-data-studio-bootstrap.yml` and some tests still treat `deploy/mysql-init/010_schema.sql` as the single source of truth

As a result, the current layout is difficult to review, difficult to evolve safely, and easy to misunderstand.

---

## 2. Goals

### 2.1 Primary goals

- Split MySQL bootstrap assets by owning subsystem and execution model
- Remove the implicit "one directory contains everything" contract from local compose wiring
- Replace the monolithic `deploy/mysql-init/010_schema.sql` bundle with explicit community source files
- Make Nacos and XXL-JOB bootstrap assets live next to their actual consumers
- Keep bootstrap flows idempotent so they work for both fresh and existing local volumes

### 2.2 Non-goals

- Change the functional schema design of community, IM, Nacos, or XXL-JOB
- Introduce a new migration framework for application runtime upgrades
- Redesign replication bootstrap behavior in `deploy/scripts/bootstrap-mysql-replication.sh`
- Support a long-lived dual layout where both the old and new directory structures remain active

---

## 3. Current-state facts

### 3.1 Main MySQL init is too implicit

`deploy/compose.infra.mysql.yml` currently mounts the entire `deploy/mysql-init/` directory into `/docker-entrypoint-initdb.d`. This makes execution depend on filename ordering inside a mixed-purpose directory.

### 3.2 Community schema is a hidden bundle

`deploy/mysql-init/010_schema.sql` contains multiple internal sections such as shared schema, demo metadata, identity, growth, content, social, message, IM, search, and seed data. The file header says it is generated by merging source files, but those source files do not exist in the repository as editable assets.

### 3.3 Nacos already behaves like an isolated subsystem

`deploy/compose.infra.nacos.yml` runs a dedicated `nacos-db-bootstrap` container that imports `deploy/mysql-init/030_nacos_schema.sql` through a dedicated script. This means the file is conceptually Nacos-owned already; only the directory placement is misleading.

### 3.4 XXL-JOB is still coupled to mysql-primary first boot

`deploy/compose.infra.xxl-job.yml` currently starts `xxl-job-admin-*` once `mysql-primary` is healthy. The actual `xxl_job` schema and local seed data are still created indirectly through mysql-primary's init directory, not through an XXL-JOB-specific bootstrap service.

### 3.5 Mock Data Studio still depends on the old bundle

`deploy/compose.infra.mock-data-studio-bootstrap.yml` currently replays `deploy/mysql-init/010_schema.sql` into the primary database. `tools/mock-data-studio/test/batch-repository.test.mjs` also reads that file directly. This means the old bundle is not only a first-boot artifact; it is also a cross-service compatibility contract.

---

## 4. Options considered

### 4.1 Option A: Keep the current directory and only improve comments

Pros:

- Lowest implementation cost
- No compose rewiring required

Cons:

- Preserves the hidden coupling
- Keeps the monolithic community schema bundle in place
- Does not align assets with their actual subsystem owners

### 4.2 Option B: Keep `deploy/mysql-init/` but add real source files under it

Pros:

- Improves community schema maintainability
- Lower risk than a full topology cleanup

Cons:

- Still keeps Nacos and XXL-JOB inside a misleading shared directory
- Still relies on the main mysql init mount as the top-level coordination mechanism
- Only partially fixes the ownership problem

### 4.3 Option C: Fully refactor bootstrap assets by subsystem and execution flow

Pros:

- Cleanest ownership boundaries
- Makes compose wiring explicit
- Lets community, Nacos, and XXL-JOB evolve independently
- Eliminates the misleading monolithic bootstrap directory

Cons:

- Requires coordinated changes across compose files, scripts, tests, and docs
- Needs careful migration of the Mock Data Studio source-of-truth contract

### 4.4 Recommendation

Adopt Option C.

The repository has already moved toward explicit layered compose topology. This refactor should continue that direction and make bootstrap assets match the same level of clarity.

## 5. Proposed filesystem layout

```text
deploy/mysql/
  conf/
    primary.cnf
    replica.cnf

  primary-init/
    001_create_databases.sh

  community/
    001_bootstrap.sh
    010_schema_shared.sql
    011_schema_demo_metadata.sql
    020_schema_identity.sql
    030_schema_growth_reward.sql
    031_schema_growth_wallet.sql
    032_schema_growth_market.sql
    033_schema_growth_task.sql
    040_schema_content_core.sql
    041_schema_content_compat.sql
    050_schema_social.sql
    060_schema_message.sql
    070_schema_im_core.sql
    080_schema_search.sql
    090_seed_identity.sql

  nacos/
    001_bootstrap.sh
    010_schema.sql

  xxl-job/
    001_bootstrap.sh
    010_schema.sql
    020_seed_local.sh
```

### 5.1 Layout rules

- `deploy/mysql/primary-init/` contains only assets that belong to mysql-primary first boot
- `deploy/mysql/community/` contains only community and `im_core` schema/seed source files
- `deploy/mysql/nacos/` contains only Nacos bootstrap assets
- `deploy/mysql/xxl-job/` contains only XXL-JOB bootstrap assets
- Number prefixes define deterministic execution order within each subsystem

## 6. Execution model

### 6.1 mysql-primary first boot

`deploy/compose.infra.mysql.yml` will stop mounting an entire directory into `/docker-entrypoint-initdb.d`. Instead, it will mount only the files needed for mysql-primary bootstrap in explicit order:

- `deploy/mysql/primary-init/001_create_databases.sh`
- `deploy/mysql/community/*.sql` source files, each mounted individually with an ordered filename under `/docker-entrypoint-initdb.d`

This keeps the MySQL image's native first-boot behavior, but removes the implicit mixed-directory contract.

### 6.2 community replay for Mock Data Studio bootstrap

`deploy/compose.infra.mock-data-studio-bootstrap.yml` will stop importing `/bootstrap/010_schema.sql`. Instead, it will invoke `deploy/mysql/community/001_bootstrap.sh`, which replays the ordered community schema and seed files explicitly against the primary database.

This keeps replay behavior while switching the source of truth from a bundle file to the real community asset set.

### 6.3 Nacos bootstrap

`deploy/compose.infra.nacos.yml` will keep `nacos-db-bootstrap`, but its mounted assets move from `deploy/mysql-init/030_nacos_schema.sql` to `deploy/mysql/nacos/`.

`deploy/mysql/nacos/001_bootstrap.sh` becomes the only Nacos DB entrypoint and is responsible for:

- ensuring the database exists
- ensuring runtime grants exist
- importing `deploy/mysql/nacos/010_schema.sql` when needed

### 6.4 XXL-JOB bootstrap

`deploy/compose.infra.xxl-job.yml` will gain an `xxl-job-db-bootstrap` one-shot service. It becomes the explicit DB gate for XXL-JOB.

`deploy/mysql/xxl-job/001_bootstrap.sh` is responsible for:

- ensuring the `xxl_job` database exists
- ensuring the runtime user and grants exist
- importing `deploy/mysql/xxl-job/010_schema.sql`
- applying `deploy/mysql/xxl-job/020_seed_local.sh`

`xxl-job-admin-1` and `xxl-job-admin-2` will depend on `xxl-job-db-bootstrap: service_completed_successfully` instead of only waiting for `mysql-primary: service_healthy`.

## 7. Community schema split details

### 7.1 Source-of-truth change

The old combined file `deploy/mysql-init/010_schema.sql` will be removed. The new source of truth becomes the ordered files under `deploy/mysql/community/`.

### 7.2 Split boundaries

The split should follow responsibility, not line count:

- `010_schema_shared.sql`: outbox and HTTP idempotency
- `011_schema_demo_metadata.sql`: `demo_*` and other Mock Data Studio metadata tables that must exist on fresh volumes
- `020_schema_identity.sql`: user/auth-refresh identity tables
- `030_schema_growth_reward.sql`: score log, reward accounts, ledgers, grant records
- `031_schema_growth_wallet.sql`: wallet accounts, wallet transactions, wallet ledger-style entries, recharge/withdraw/transfer/admin actions
- `032_schema_growth_market.sql`: market listing, inventory, delivery, order, dispute, address, shipment
- `033_schema_growth_task.sql`: task template, progress, task event log, and related seed rows
- `040_schema_content_core.sql`: post/comment/category/tag/report/moderation/bookmark/subscription tables
- `041_schema_content_compat.sql`: idempotent compatibility upgrades and index backfills for content tables
- `050_schema_social.sql`: like/follow/block tables
- `060_schema_message.sql`: station-message schema
- `070_schema_im_core.sql`: `im_core` tables
- `080_schema_search.sql`: search-owned DB assets, even if currently minimal
- `090_seed_identity.sql`: dev/demo-only users

### 7.3 Why content is split into core + compat

The current content section mixes canonical DDL with replay-safe compatibility upgrades. Keeping them together makes review difficult. Splitting them preserves behavior while making the intent readable:

- core file = desired schema
- compat file = safe patch-up logic for older local volumes

### 7.4 Why task seed data lives with task schema

`task_template` seed rows are configuration data tightly coupled to the task subsystem. They should live beside the task schema rather than being buried in a broader growth file.

## 8. Script responsibility changes

### 8.1 `001_create_databases.sh`

After the refactor, `deploy/mysql/primary-init/001_create_databases.sh` should be narrowed to the main product databases and users that belong to mysql-primary's first-boot responsibility:

- `community`
- `im_core`
- shared local dev users directly required by those schemas, including Mock Data Studio DB grants

It should no longer be responsible for Nacos or XXL-JOB bootstrap concerns.

### 8.2 subsystem bootstrap scripts own their own DB lifecycle

- `deploy/mysql/nacos/001_bootstrap.sh` owns Nacos database creation, grants, and schema import
- `deploy/mysql/xxl-job/001_bootstrap.sh` owns XXL-JOB database creation, grants, schema import, and local seed import

This removes the current "central script creates every database for every subsystem" model.

## 9. Testing and validation strategy

### 9.1 Static validation

Run shell syntax checks on all new bootstrap scripts:

- `bash -n deploy/mysql/primary-init/001_create_databases.sh`
- `bash -n deploy/mysql/community/001_bootstrap.sh`
- `bash -n deploy/mysql/nacos/001_bootstrap.sh`
- `bash -n deploy/mysql/xxl-job/001_bootstrap.sh`
- `bash -n deploy/mysql/xxl-job/020_seed_local.sh`

### 9.2 Compose validation

Run compose rendering for the full local stack and confirm:

- explicit mounts resolve correctly
- the new `xxl-job-db-bootstrap` service is present
- service dependencies are valid

### 9.3 Source-of-truth test migration

Tests that currently parse `deploy/mysql-init/010_schema.sql` must move to the new community source layout. For Mock Data Studio, tests should verify:

- expected metadata DDL is present in `deploy/mysql/community/011_schema_demo_metadata.sql`
- bootstrap replay order is correct in `deploy/mysql/community/001_bootstrap.sh`

The test suite should stop asserting the existence of a single monolithic schema bundle.

### 9.4 Runtime validation

On a fresh local bring-up, verify:

- mysql-primary first-boot init succeeds
- `nacos-db-bootstrap` completes successfully
- `xxl-job-db-bootstrap` completes successfully
- `mock-data-studio-db-bootstrap` completes successfully
- Nacos and XXL-JOB services start without schema race conditions

### 9.5 Existing-volume validation

Without deleting volumes, rerun the bootstrap sidecars and confirm replay remains safe. This is required because the refactor is intended to preserve idempotent local recovery behavior.

## 10. Migration plan

1. Create the new `deploy/mysql/` subtree
2. Move and split community schema assets into ordered source files
3. Add community, Nacos, and XXL-JOB bootstrap scripts
4. Rewire `deploy/compose.infra.mysql.yml` to explicit mounts
5. Rewire `deploy/compose.infra.mock-data-studio-bootstrap.yml` to use community bootstrap
6. Rewire `deploy/compose.infra.nacos.yml` to use `deploy/mysql/nacos/`
7. Add `xxl-job-db-bootstrap` and update `deploy/compose.infra.xxl-job.yml`
8. Migrate tests and docs away from old paths
9. Remove `deploy/mysql-init/` and the old `deploy/scripts/bootstrap-nacos-db.sh`
10. Run static, compose, and runtime verification

This refactor should be landed as a single coherent change set rather than a partial long-lived compatibility layer.

## 11. Risks and mitigations

### 11.1 Risk: startup ordering regressions

If service dependencies are incomplete, Nacos or XXL-JOB may start before schema/bootstrap completion.

Mitigation:

- add explicit bootstrap sidecars
- use `service_completed_successfully` where appropriate
- verify with actual compose bring-up

### 11.2 Risk: hidden path dependencies remain

Some tests or docs may still reference `deploy/mysql-init/`.

Mitigation:

- run repository-wide searches for old paths before finalizing
- treat zero remaining references as a completion gate

### 11.3 Risk: replay behavior diverges from first-boot behavior

If community bootstrap replay and mysql-primary first-boot mounts use different file sets or order, behavior can drift.

Mitigation:

- keep community files as the single source of truth
- ensure both first-boot and replay flows consume the same ordered file set

---

## 12. Success criteria

This refactor is successful when all of the following are true:

- `deploy/mysql-init/` no longer exists
- community, Nacos, and XXL-JOB bootstrap assets are separated by subsystem
- mysql-primary no longer mounts one mixed-purpose init directory
- XXL-JOB has an explicit DB bootstrap gate like Nacos
- Mock Data Studio no longer depends on a monolithic `010_schema.sql`
- tests and docs reference the new structure only
- local bootstrap remains idempotent for both fresh and existing volumes

---

## 13. Out of scope for implementation planning

The following topics are intentionally excluded from this refactor:

- changing business table semantics
- introducing Flyway/Liquibase
- restructuring non-MySQL infra assets under `deploy/`
- changing MySQL replication topology or promotion workflows
